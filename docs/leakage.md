# Leakage 與混淆

這份公開資料的評估數字**不可當成真實臨床表現**。以下四層問題,前三層可以量測或處理,
第四層無解。程式碼裡的相關開關(`DEDUP` / `MASK`)都指回這份文件。

---

## 第 1 層:完全重複的影像(已處理)

`config.DEDUP = "exact"`(或 `True`)

6323 個檔案裡,只有 **1536 張位元組不重複**的影像 —— 平均每張被複製約 4 次,
最多的一張出現 18 次。

隨機 split 會讓同一張圖的複製品同時落在 train 與 test。實測 `train ∩ test`
有 **636 張位元組完全相同**的影像,模型用背的就滿分:

| | 去重前 | 去重後 |
|---|---|---|
| test macro AUROC | **0.9975** | 0.94 |
| F0 recall | 1.000 | 0.97 |
| QWK | 0.993 | 0.83 |

`_dedup_indices(mode="exact")` 依 md5 分組,每組只留第一個出現的 index
(`ImageFolder.samples` 已按路徑排序 → 決定性,不受檔案系統列舉順序影響)。

**附註:** 1536 個 hash 全部只對應單一類別,**沒有跨類別重複** → 標籤本身沒有矛盾。

## 第 2 層:近重複的影像(可選處理)

`config.DEDUP = "perceptual"`

exact 去重後的 1536 張裡,還有約 **148 張是「重新壓縮 / 縮放過」的近重複**
(dHash 相同、位元組不同)。md5 抓不到這種。

**F0 特別多:317 → 199。** 這正是 F0 recall 一直偏高(0.97~1.00)的可疑來源。

| dedup | 總數 | F0 | F1 | F2 | F3 | F4 |
|---|---|---|---|---|---|---|
| exact | 1536 | 317 | 296 | 308 | 308 | 307 |
| **perceptual** | **1388** | **199** | 288 | 308 | 308 | 285 |

### 為什麼只在同類別內合併

超音波影像都是「黑背景 + 扇形探頭區」,低頻結構相近 → 64-bit 的 dHash 偶爾會讓
兩張**其實不同組織**的圖(特別是 F0 / F4)撞到同一個雜湊。實測有 7 組跨類別碰撞,
但那些是**雜湊碰撞不是真重複**。所以 `_perceptual_hash` 的 key 帶上 label,
只在同類別內合併,避免誤刪。

### 效果:不大,但排除了一個疑慮

| 指標 | exact CV | perceptual CV |
|---|---|---|
| macro AUROC | 0.936 ± 0.011 | 0.934 ± 0.013 |
| balanced accuracy | 0.742 ± 0.026 | 0.754 ± 0.030 |
| **F0 recall** | **0.972 ± 0.032** | **0.940 ± 0.028** |

F0 recall 只掉 0.03,而且仍是最高的一類。**近重複確實有灌一點水,但 F0 本質上就好認** ——
「模型只是在認 F0 的複製品」這個假設不成立。

per-class 的 std 也普遍下降(F1 從 ±0.174 → ±0.094),perceptual 版的 CV 更穩定,
**應該當作正式基準**。

## 第 3 層:來源混淆(source confounding)🚨

比 leakage 更嚴重,而且**無法從測試分數看出來**。

perceptual 去重後,**檔名前綴(= 資料來源)與類別幾乎完全綁定**:

| 類別 | n | 主要前綴 |
|---|---|---|
| **F0** | 199 | **`a` 佔 198 張(99.5%)** |
| F1 | 288 | `ct` 45 + 雜 |
| F2 | 308 | `ct` 128 + 雜 |
| F3 | 308 | `ct` 118 + 雜 |
| F4 | 285 | `G`/`F`/`J`/`I` 雜(無 `ct`、無 `a`) |

**F0 ≡ `a` 來源**;`ct` 前綴(926 個 `.png`)只出現在 F1~F3。

這份資料是**多來源合併**的。模型只要學會「認來源」(機器型號、前處理、扇形幾何、
增益設定)就能拿到大部分分數 —— 而分數本身就被混淆灌水了,看不出來。

**也無法用子群比較檢驗**:非 `a` 的 F0 只剩 1 張,沒有對照組。

### 怎麼測:遮罩消融

`config.MASK`。把影像的一部分塗黑後重訓,直接證明模型靠什麼吃飯:

| 遮罩 | frac | 保留什麼 | 判讀 |
|---|---|---|---|
| `center` | **0.9** | 塗黑中央,組織只剩 **1.9%** | 表現仍高 → 靠來源假影 🚨 |
| `periphery` | **0.6** | 塗黑邊緣,保留 **69%** 組織 | 表現仍高 → 真的在讀組織 ✅ |

**兩種遮罩必須用不同的 frac**(在 200 張隨機影像上實測):

| frac | center 後組織剩餘 | periphery 後保留的組織 |
|---|---|---|
| 0.6 | — | 69% |
| 0.7 | 16.2% ← 清不乾淨 | 84% |
| 0.85 | 4.6% | — |
| **0.9** | **1.9%** ✅ | 98% ← 等於沒遮 |

判讀時比的是**「超越隨機的本事」**(`balanced_accuracy − 0.2`),不是 balanced accuracy
本身 —— 後者會被 0.2 的地板灌水。

### 輔證:Grad-CAM(`explain.py`)

算「邊緣佔比」= 熱圖能量落在外框 20% 的比例。高 = 盯著文字標註 / 探頭刻度 / 扇形邊緣。
比消融弱(只說「看哪裡」,不說「沒有組織能不能活」),但便宜。

## 第 4 層:patient-level leakage(無解)

檔名是「字母前綴 + 流水號」(`a1000.jpg` / `I2079.jpg` / `z9945.jpg`…),
**前綴不對應病人,沒有病人 ID** → 做不到 patient-level split。

即使去重,同一位病人的不同切面 / 不同幀仍可能分散在不同 split。

**因此所有數字都仍然偏樂觀,是上限而非估計值。**

真實的 patient-level 評估留給未來的臨床資料 —— 屆時把 `dataset._dedup_indices`
換成依病人 ID 分組的 `StratifiedGroupKFold` 即可,其餘流程不用動。
